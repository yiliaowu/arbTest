# 故障复盘：QMT Socket 全量行情推送、死锁与数据解包避坑指南

**文档状态**：核心架构排雷记录
**最后更新**：2026-04-27
**核心贡献**：打破了关于 QMT 机制的多个伪科学误区，确立了高频 Socket 行情穿透的绝对正确架构。

---

## 一、 四大伪科学误区与真相澄清

在开发 `qmt_socket_server.py` 与 `LOF02` 主程序联调的过程中，由于行情时断时续、长连接“静默死机”，我们曾陷入过几个严重的逻辑误区。特此澄清：

### ❌ 误区 1：QMT 存在“UI 懒加载”限制
*   **错误认知**：认为 QMT 为了节省内存，`get_full_tick` 只能获取到当前软件 UI 界面上正在显示的股票，不在屏幕上的标的会被底层静默拦截。
*   **真相**：**纯属无稽之谈！** QMT 内置引擎的 `get_full_tick` 是绝对的全量内存快照获取。只要向系统发送了查询指令，无论标的是否在自选股或屏幕上，底层都会如实返回。如果没返回数据，一定是 Python 层的代码抛出了异常导致线程崩溃。

### ❌ 误区 2：需要使用 `subscribe_quote` 强行激活
*   **错误认知**：为了打破所谓的“懒加载”，试图在 QMT 内嵌策略中调用 `ContextInfo.subscribe_quote` 强制订阅行情。
*   **真相**：`subscribe_quote` 是外置独立版 `miniQMT (xtquant)` 的专属 API。在 QMT 内置 C++ 策略引擎中强行调用该函数，属于跨环境乱用 API，反而会导致不可预知的环境污染或静默报错。

### ❌ 误区 3：批量查询 `get_full_tick([list])` 容易崩溃，应该用 for 循环单查
*   **错误认知**：认为传入长列表时，如果遇到一个“毒药代码”（如退市或错误代码），会导致整个查询崩溃，因此退化为 `for` 循环逐个查询。
*   **真相**：**这是极其低效的倒退！** 批量查询之所以崩溃，**根本不是因为 C++ 底层查不到数据，而是因为 Python 层在处理返回的脏数据时（见下方解包陷阱），触发了类型错误！** C++ 引擎设计就是为了批量吞吐，直接传入全量列表才是性能最优、最正确的做法。

### ❌ 误区 4：没有日志输出就是 QMT 没推数据
*   **错误认知**：主程序启动后，黑窗口没有打印闪电图标 `⚡`，就认为是 QMT 没发数据。
*   **真相**：是外部主程序（`LOF02`）中“新浪兜底爬虫”的 0 秒抢跑，外加全局日志标记的 Bug，把正确到达的 QMT 数据给静默拦截了。

## 二、 真正的致命杀手（核心 Bug 揭秘）

导致长连接瞬间哑火、数据流中断的真正元凶是以下三个底层技术陷阱：

### 💀 致命杀手 1：Tuple 拼接陷阱 (TypeError 崩溃)
*   **案发现场**：在提取五档盘口时使用了 `ap = tick.get('askPrice') + [0]*5` 进行防御性补齐。
*   **崩溃原因**：在非交易时间或遭遇流动性极差的标的时，QMT 返回的 `askPrice` 并非标准的 `List`，而是 `Tuple`（元组），甚至可能是 `None`。Python 中 `Tuple + List` 会瞬间触发 `TypeError: can only concatenate tuple to tuple`。
*   **后果**：这导致处理到某一只基金时，推流线程当场崩溃，后续所有基金的推送被直接腰斩。
*   **解决方案**：引入 `safe_list` 强制类型转换：
    ```python
    def safe_list(val):
        if isinstance(val, (list, tuple)): return list(val) + [0]*5
        return [0]*5
    ```

### 💀 致命杀手 2：端口霸占幽灵 (Address already in use)
*   **案发现场**：修改了 QMT 策略代码后，点击“保存并运行”，发现新代码就是不生效。
*   **崩溃原因**：QMT 重新运行策略时，默认**不会杀掉旧的 Socket 线程**。新代码启动时发现 `8888` 端口被占用，触发异常，但原来的代码里写了 `except Exception: pass`（或 `return`），导致新代码悄悄退出，后台依然运行着那个会崩溃的旧版本。
*   **解决方案**：遇到死寂时，**彻底关闭并重启 QMT 客户端**释放端口，并在 `server.bind()` 的异常捕获中加入高亮打印，杜绝静默失败。

### 💀 致命杀手 3：回调死锁 (Deadlock)
*   **案发现场**：`LOF02` 的 Socket 客户端在收到第一笔推送后，瞬间卡死，再也不接收后续数据。
*   **崩溃原因**：接收线程 `_recv_loop` 使用 `with self.lock:` 锁住字典更新数据，同时触发回调 `on_price_update`。而外部主程序在回调中，立刻调用 `client.get_order_book()` 获取盘口，该函数**试图再次获取同一把锁**。由于使用的是普通的 `threading.Lock()`（不可重入锁），线程自己把自己锁死了。
*   **解决方案**：将普通的 `Lock()` 升级为可重入锁 `threading.RLock()`，允许同一线程在回调嵌套中反复进出安全区。

## 三、 结论与最佳实践

1. **绝对信任 QMT 的全量查询**：`get_full_tick` 是极其可靠的底层接口。只要传入规范的列表，它就会如实返回快照。
2. **警惕 Python 的动态类型**：在解析底层 C++ 引擎传回的数据时，绝不能想当然地认为是列表，必须做好 `None` 和 `Tuple` 的防御性校验。
3. **慎用普通 Lock**：在包含 Callback (回调机制) 的异步网络编程中，凡是暴露给外部调用的公共数据结构，一律使用 `RLock`。
